01 Selected work

01

Cinna

Private

Solo · Flutter client + Django backend · 2024–2025

Problem Public-transit riders are stuck with schedules that drift from reality and clunky paper-or-card ticketing — no single place to see where the bus actually is and pay for the ride.

Approach A real-time transit companion: a Flutter/Riverpod client over a Django REST API that ingests live GTFS-RT feeds, answers geospatial queries with PostGIS, and caches hot data in Redis — plus an in-app wallet for ticketing.

  • GTFS-RT ingestion behind a Redis cache so live vehicle positions are served without hammering Postgres.
  • PostGIS for spatial work — nearest-stop, route geometry, “arriving near me”.
  • Riverpod to keep UI state predictable across real-time streams.
  • A wallet abstraction that keeps ticketing logic independent of the payment provider.

Outcome A working end-to-end system (live tracking → trip planning → ticket) running on a geospatial backend. Private repo — walkthrough/demo available on request.

02

Aristophanes

Private

Solo · Native iOS (Flutter) · architecture-first

Problem Mobile apps rot when business logic is welded to the framework. I wanted a codebase whose core could outlive any SDK churn and stay genuinely testable.

Approach A native iOS app built to a strict Ports-and-Adapters (hexagonal) architecture: the domain sits in the centre, frameworks and I/O are adapters at the edges, and every significant decision is written down as an Architecture Decision Record.

  • Domain logic depends on nothing — UI, storage and platform are plug-in adapters.
  • ADRs capture the “why” behind each call, so the reasoning is reviewable, not folklore.
  • A pure core that can be unit-tested without a simulator.

Outcome A framework-independent, test-first core with a documented decision log. Private repo — code walkthrough on request.

Plutarch

Public

Solo · Next.js + PostgreSQL · open source

Problem Greek dictionaries online are slow and shallow — hard to scan, weak on cross-references, painful to explore a word's neighbourhood.

Approach A modern Greek dictionary & language-exploration web app — fast lookup, structured entries, and cross-references — built on Next.js + TypeScript over a PostgreSQL data layer.

  • Structured, normalised entries so meanings and relations are first-class, not free text.
  • Server-rendered Next.js for fast, shareable, search-friendly pages.

Outcome Public, open-source, and still growing. Source on GitHub.
